# Impeccable — contexte design (Kasistar)

## Design Context

### Users

- **Qui** : joueurs francophones qui font une **devinette quotidienne** (célébrité / jeu de mots à partir d’une image-indice).
- **Contexte** : sessions courtes (quelques minutes), souvent mobile ou bureau ; le **jour de jeu** suit **Europe/Paris** (`src/lib/puzzle-constants.ts`).
- **Objectif** : soumettre un nom, recevoir un retour clair (succès / erreur), éventuellement consulter les **archives** (calendrier).
- **État technique** : données puzzle en dépôt (`content/puzzles/`, `public/puzzles/`), validation côté serveur (`submitGuess`).

### Brand Personality

- **Voix** : français, direct, ton léger (ex. footer « Une énigme par jour »).
- **En trois mots (proposition)** : **ludique**, **claire**, **quotidienne**.
- **Nom** : **Kasistar** — identité visuelle en cours (logo favicon dans le header, typo display récente).
- **À affiner avec l’équipe** : référence culturelle (pure divertissement vs. « jeu de mots intello »), niveau de second degré.

### Aesthetic Direction

- **Stack UI** : React 19, TanStack Start + Router, **Tailwind** v4, **shadcn** (`components.json` : `baseColor: neutral`, `cssVariables: true`), **Base UI** pour composants accessibles.
- **Typographie** : **Sora** (corps / UI), **Bricolage Grotesque** (titres / marque via `--font-heading`) — chargées depuis Google Fonts dans `src/styles.css`.
- **Couleur** : accent **terracotta** (OKLCH `~hue 28`) pour `--primary`, fonds légèrement **chauds** ; sémantiques destructive / info / success / warning en OKLCH alignées. Voir `:root` / `.dark` dans `src/styles.css`.
- **Thème** : **clair + sombre** avec persistance `localStorage` (`ThemeToggle`, script d’init dans `__root.tsx`).
- **Layout** : header sticky avec bordure + **backdrop-blur**, `max-w-4xl` pour le contenu principal, `max-w-4xl` header, footer discret.
- **Feedback** : micro-sons (hooks `use-sound`) sur succès / erreur ; états async via `@bearstudio/ui-state` sur la page d’accueil.

### Design Principles

1. **Une action principale** : deviner doit rester évidente ; formulaire et boutons lisibles, pas de surcharge.
2. **Clarté des états** : chargement, erreur, absence de puzzle, succès — chaque état doit être compris sans jargon.
3. **Cohérence des tokens** : couleurs et rayons via variables CSS (`--radius`, `--primary`, etc.) ; éviter les couleurs en dur hors du système.
4. **Accessibilité de base** : lien « Aller au contenu », focus visible, contrastes neutres ; préférer les composants accessibles (Base UI / patterns shadcn).
5. **Frugalité motion** : animations et sons au service du feedback, pas pour encombrer ; respecter `prefers-reduced-motion` quand on ajoute du mouvement (à vérifier sur les écrans existants).

## Questions ouvertes (à compléter avec le produit)

Ces points ne sont pas déductibles du seul code :

1. **Public cible** : âge, habitudes (uniquement mobile ? partage social ?).
2. **Émotion visée** : plutôt **calme / concentré**, **excité**, **nostalgie** (presse people) ?
3. **Références** : sites ou apps dont « l’ambiance » colle à Kasistar — et ce qu’il faut **éviter** (anti-références).
4. **Accessibilité** : objectif WCAG (A/AA), besoins particuliers connus ?
5. **Couleurs** : une couleur d’accent ou de marque à imposer ou à bannir ?
